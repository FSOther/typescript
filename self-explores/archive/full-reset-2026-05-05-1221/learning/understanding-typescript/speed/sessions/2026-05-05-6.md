### L234-L235 — Typing Component Props (DEEP_DIVE)

**80/20:** Định nghĩa `interface Props` hoặc `type Props` → dùng làm type annotation cho parameter.

**Must-know:**
```typescript
// Define props shape
interface HeaderProps {
  image: {
    src: string;
    alt: string;
  };
}

// Type the parameter
function Header({ image }: HeaderProps) {
  return <header><img {...image} /></header>
}

// Usage — TypeScript enforces shape
<Header image={{ src: goalsImg, alt: "Course goals" }} />
```

**Active Recall:**
1. Q: Cách type props trong React component function? → A: `function MyComp({ prop }: MyProps)` — interface/type as parameter type
2. Q: Tại sao KHÔNG dùng `React.FC<Props>`? → A: Modern TS recommended: đơn giản hơn, không cần import FC, direct parameter typing

### L239 — Typing `useState` (DEEP_DIVE)

**80/20:** `useState<T>(initialValue)` = explicit generic OR let TypeScript infer from initial value.

**Must-know:**
```typescript
type Goal = { id: number; title: string; description: string };

// Option 1: Infer from initial value (when type is clear)
const [goals, setGoals] = useState<Goal[]>([]);
// OR let TS infer if initial value is typed:
// const [goals, setGoals] = useState([] as Goal[])

// Option 2: Explicit generic (when initial value is ambiguous)
const [count, setCount] = useState<number>(0);
```

**Active Recall:**
1. Q: `useState([])` vs `useState<Goal[]>([])` — difference? → A: Without generic: TS infers `never[]`; with generic: TS knows it's `Goal[]`
2. Q: Functional state update pattern? → A: `setGoals(prev => prev.filter(g => g.id !== id))` — use when new state depends on old

### L242 — `useRef` Typing (DEEP_DIVE)

**80/20:** `useRef<HTMLInputElement>(null)` — explicit generic needed because TS can't infer from `null` + JSX connection.

**Must-know:**
```typescript
import { useRef } from 'react';

function NewGoal() {
  const goalRef = useRef<HTMLInputElement>(null);
  const summaryRef = useRef<HTMLInputElement>(null);
  
  function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();
    const enteredGoal = goalRef.current!.value;     // ! = non-null assertion (safe after mount)
    const enteredSummary = summaryRef.current!.value;
  }
  
  return (
    <form onSubmit={handleSubmit}>
      <input ref={goalRef} type="text" />
      <input ref={summaryRef} type="text" />
    </form>
  );
}
```

**Active Recall:**
1. Q: `useRef<HTMLInputElement>(null)` — tại sao cần explicit generic? → A: TS infers from `null` → type is `null`; generic cho TS biết ref sẽ hold HTMLInputElement
2. Q: `goalRef.current!.value` — `!` ở đây có an toàn không? → A: Có — inside event handler, component đã mounted → ref đã được attached

### L250 — Express.js Types (DEEP_DIVE)

**80/20:** `@types/express` = Express type definitions. Request/Response/NextFunction types available.

**Must-know:**
```typescript
import express from 'express';
import { Request, Response, NextFunction } from 'express';

const app = express();
app.use(express.json());  // parse JSON body

// Route — TypeScript infers req/res types inline:
app.get('/todos', (req, res) => {
  res.json({ todos: [] });
});

// Standalone function — must annotate explicitly:
function handleGet(req: Request, res: Response) {
  res.json({ todos: [] });
}

// Error middleware — 4 params:
function errorHandler(err: Error, req: Request, res: Response, next: NextFunction) {
  res.status(500).json({ message: 'An error occurred' });
}
```

**Active Recall:**
1. Q: Khi viết route handler inline vs standalone function — type annotation khác gì? → A: Inline: TS infers `Request`/`Response` automatically; Standalone: phải explicit annotate
2. Q: `request.body` type là gì trong Express? → A: `any` — Express không biết request body shape của bạn

### L251-L253 — Full CRUD REST API (STUDY)

**80/20:** TypeScript catch type errors (ID string vs number, optional return values). Import extension `.js` even for `.ts` files when compiling.

**Key patterns:**
```typescript
// Define data types
interface Todo {
  id: number;
  text: string;
}

let todos: Todo[] = [];

// Type-safe function
function getTodo(id: number): Todo {
  const todo = todos.find(t => t.id === id);
  if (!todo) throw new Error('Todo not found');
  return todo;  // guaranteed Todo (not Todo | undefined)
}

// URL param is always string → must convert
app.get('/todos/:id', (req, res) => {
  const todo = getTodo(+req.params.id);  // + converts string → number
  res.json({ todo });
});
```

**IMPORTANT:** When compiling, imports must use `.js` extension even for `.ts` source files:
```typescript
import { getTodos } from './data.js';  // NOT data.ts (even though file is data.ts)
```

**Active Recall:**
1. Q: `req.params.id` type → A: `string` always (URL is text) → convert with `+req.params.id` or `parseInt()`
2. Q: Tại sao import `./data.js` khi file là `data.ts`? → A: Node.js chạy compiled JS — TS compiler maps `data.ts` → `data.js` — import phải point đến compiled output

### L254-L255 — Native Node TS + Type Stripping (STUDY)

**80/20:** Node v22.6+ can run `.ts` directly with `--experimental-strip-types` (strips annotations, doesn't compile).

**Key distinction:**
| Feature | Type Stripping | Type Transformation |
|---|---|---|
| Interfaces/types | ✅ Supported (dropped) | ✅ |
| Type annotations | ✅ Supported (dropped) | ✅ |
| Enums | ❌ NOT supported | ✅ (needs `--experimental-transform-types`) |
| Decorators | ❌ NOT supported | ✅ |
| Performance | Faster | Slower |

**Config for native Node TS:**
```json
{
  "compilerOptions": {
    "allowImportingTsExtensions": true,
    "noEmit": true,
    "verbatimModuleSyntax": true
  }
}
```

With `verbatimModuleSyntax: true`:
```typescript
import { type Request, type Response } from 'express';  // `type` required for type-only imports
```

**Active Recall:**
1. Q: `verbatimModuleSyntax: true` forces what? → A: `type` keyword required on all type-only imports — helps Node strip them correctly
2. Q: `--experimental-strip-types` vs `--experimental-transform-types` → A: Strip = drop annotations only (fast, no enums); Transform = full compile (slower, all features)
