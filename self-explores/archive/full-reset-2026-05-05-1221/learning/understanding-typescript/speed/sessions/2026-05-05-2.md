### L69 — Creating a First Class (STUDY)

**80/20:** Classes trong TS: properties declared outside constructor (không phải trong constructor như JS). Constructor shortcut với `public`.

**Must-know:**
```typescript
class User {
  name: string;  // declare field ở đây (TS-specific)
  age: number;
  constructor(n: string, a: number) {
    this.name = n;
    this.age = a;
  }
}
```
- TypeScript yêu cầu declare properties outside constructor body
- Uninitialized properties phải được assign trong constructor (hoặc TS error)

**Active Recall:**
1. Q: Khác nhau khai báo class property giữa JS và TS? → A: JS: `this.name = value` trong constructor; TS: khai báo `name: string` outside constructor trước
2. Q: Nếu property không có initial value và không assign trong constructor → lỗi gì? → A: "Property not definitely assigned" error
3. Q: Convention đặt tên class? → A: PascalCase (UpperCamelCase) — không bắt buộc nhưng standard

### L70 — Constructor Shortcut (STUDY)

**80/20:** Thêm `public`/`private` trước param trong constructor → TS tự tạo property và gán value. Shortcut cực kỳ ngắn.

**Must-know:**
```typescript
class User {
  constructor(public name: string, public age: number) {}
  // TS tự tạo this.name và this.age properties
}
const max = new User("Max", 36)  // max.name = "Max", max.age = 36
```
- Shortcut chỉ work khi dùng `public`/`private`/`protected`/`readonly`
- Không cần constructor body nếu chỉ cần assign props

**Active Recall:**
1. Q: `constructor(public name: string)` tương đương với code dài nào? → A: `name: string; constructor(n: string) { this.name = n; }`
2. Q: Phải dùng keyword gì để kích hoạt shortcut? → A: `public`, `private`, `protected`, hoặc `readonly`
3. Q: Nếu bỏ `public` trong constructor param → sao? → A: Shortcut không hoạt động — param là local variable chứ không phải property

### L71 — `public` và `private` (STUDY)

**80/20:** `public` = accessible everywhere (default). `private` = only inside class. TypeScript-specific keywords.

**Must-know:**
- `public name: string` = readable + writable từ outside
- `private age: number` = only accessible inside class methods
- JS equivalent: `#age` (different syntax, same concept)
- `protected` = like private but accessible in subclasses

**Active Recall:**
1. Q: Default visibility của property là gì nếu không viết keyword? → A: `public` (implicit)
2. Q: `private age` — có thể access `user.age` từ ngoài không? → A: Không — TS error "Property 'age' is private"
3. Q: `public` và `private` có tồn tại trong compiled JS không? → A: Không — TS-only, stripped at compile time

### L72 — `readonly` (STUDY)

**80/20:** `readonly` = có thể đọc, không thể reassign. Nhưng có thể mutate (push vào array).

**Must-know:**
```typescript
class User {
  public readonly hobbies: string[] = [];
}
user.hobbies = ["new"]  // ERROR: readonly
user.hobbies.push("sports")  // OK: mutating, not reassigning
```
- readonly ≠ frozen — array vẫn có thể bị mutate
- Combine với `public`: `public readonly name: string`

**Active Recall:**
1. Q: Khác nhau `readonly` và `private`? → A: `private` = can't access outside class; `readonly` = can read, can't reassign
2. Q: `readonly arr: string[] = []` — `arr.push("x")` có lỗi không? → A: Không — push mutates, không reassign
3. Q: `readonly arr: string[] = []` — `arr = []` có lỗi không? → A: Có — đây là reassignment

### L73 — Getters (STUDY)

**80/20:** `get propName() { return ... }` = computed property. Dùng như property (không gọi `()`).

**Must-know:**
```typescript
class User {
  constructor(private firstName: string, private lastName: string) {}
  get fullName() { return `${this.firstName} ${this.lastName}` }
}
console.log(user.fullName)  // không cần ()
```
- Getter phải `return` value
- Dùng khi muốn property được tính on-the-fly từ other properties

**Active Recall:**
1. Q: Getter được call như thế nào? → A: Như property: `user.fullName`, không phải `user.fullName()`
2. Q: Tại sao dùng getter thay vì method? → A: Để access như property, tránh người dùng nhầm gọi như method
3. Q: Getter phải có gì không? → A: Phải có `return` statement

### L74 — Setters (STUDY)

**80/20:** `set propName(value) { ... }` = validate trước khi set. Gọi như assignment.

**Must-know:**
```typescript
class User {
  private _firstName = "";
  set firstName(name: string) {
    if (!name.trim()) throw new Error("Invalid name")
    this._firstName = name
  }
}
user.firstName = "Max"  // triggers setter
```
- Setter không return gì
- Phải có exactly 1 parameter
- Tên setter ≠ tên private property (thường prefix `_`)

**Active Recall:**
1. Q: Setter được gọi như thế nào? → A: Như assignment: `user.firstName = "Max"`
2. Q: Setter có bao nhiêu parameter? → A: Exactly 1 — required
3. Q: Tại sao dùng setter thay vì method? → A: Syntax cleaner như assignment; có thể validate trước khi set

### L75 — Static Properties & Methods (STUDY)

**80/20:** `static` = thuộc về class, không phải instance. Access via `ClassName.prop`.

**Must-know:**
```typescript
class User {
  static eid = "user"
  static greet() { console.log("Hello!") }
}
User.eid      // OK — no new User() needed
User.greet()  // OK
```
- Useful cho utility methods không cần instance data
- Cannot access `this.instanceProp` inside static method

**Active Recall:**
1. Q: Static property thuộc về ai? → A: Thuộc về class, không phải instance
2. Q: `new User().greet()` hay `User.greet()`? → A: `User.greet()` — static method call trên class, không phải instance
3. Q: Khi nào dùng static? → A: Utility methods không cần instance data; factory methods; shared constants

### L76 — Inheritance (STUDY)

**Must-know:**
```typescript
class Employee extends User {
  constructor(public jobTitle: string) {
    super()  // MUST call parent constructor first
```

- `extends` = standard JS feature (not TS-specific)
- `super()` required if you define constructor in child class
- Can override parent methods

**Active Recall:**
1. Q: Cần gọi gì trong constructor của subclass? → A: `super()` — gọi parent constructor
2. Q: Có thể override parent method không? → A: Có — define same method name in child class
3. Q: `extends` là JS hay TS? → A: JavaScript — standard JS feature

### L77 — `protected` Modifier (STUDY)

**80/20:** `protected` = like `private` nhưng accessible trong subclasses. Giải quyết vấn đề inheritance + private.

**Must-know:**
- `private`: only THIS class
- `protected`: this class + ALL subclasses
- `public`: everywhere

**Active Recall:**
1. Q: Khác nhau `private` và `protected`? → A: `private` = chỉ class đó; `protected` = class đó + mọi subclass
2. Q: Khi nào nên dùng `protected` thay `private`? → A: Khi subclass cần truy cập property của parent class
3. Q: Access modifiers có 3 levels là gì? → A: `private` → `protected` → `public` (từ restrictive nhất đến public nhất)

### L78 — Abstract Classes (DEEP_DIVE)

**80/20:** `abstract` = class không thể instantiate trực tiếp — chỉ dùng làm base class. TS-only feature.

**Must-know:**
```typescript
abstract class UIElement {
  constructor(protected id: string) {}
  abstract render(): void  // subclass MUST implement
}
// new UIElement("id")  // ERROR: cannot instantiate abstract class
class Button extends UIElement {
  render() { /* concrete implementation */ }
}
```
- `abstract` class: TS-only, không tồn tại trong compiled JS
- `abstract` method: không có body → subclass PHẢI implement
- Dùng khi muốn base class không thể be used standalone

**Difference abstract vs interface:**
| | Abstract Class | Interface |
|---|---|---|
| Can have implementation | Yes | No |
| Can have constructor | Yes | No |
| Can have state (props) | Yes | Properties only (no values) |
| JS representation | Yes (class in JS) | No (compiled away) |
| Multiple inheritance | No (only one extends) | Yes (implements multiple) |

**Active Recall:**
1. Q: Khi nào dùng abstract class? → A: Khi base class không nên instantiate trực tiếp, nhưng cần share implementation code với subclasses
2. Q: Abstract method có body không? → A: Không — subclass PHẢI provide implementation
3. Q: Abstract class được compile sang JS không? → A: Class được compile, nhưng `abstract` keyword bị strip

### L79-L80 — Introducing & Creating Interfaces (DEEP_DIVE)

**80/20:** Interface = define object shape (type contract). `interface` keyword. No values, no logic — chỉ types.

**Must-know:**
```typescript
interface Authenticatable {
  email: string
  password: string
  login(): void
  logout(): void
}
let user: Authenticatable = {
  email: "test@test.com",
  password: "abc",
  login() { /* ... */ },
  logout() { /* ... */ }
}
```
- Interface = TS-only feature
- Chỉ defines SHAPE, không có implementation
- Used as object types (like type aliases)

**Active Recall:**
1. Q: Interface khác class ở điểm gì cơ bản nhất? → A: Interface = chỉ shape/type definition, không có values/logic; Class = blueprint với implementation
2. Q: Interface có compile sang JS không? → A: Không — completely stripped, không tồn tại trong JS
3. Q: Interface dùng được cho variable type không? → A: Có — `let x: MyInterface = { ... }`

### L82 — Interface vs Type Alias (DEEP_DIVE) — CRITICAL: 3 Differences

**80/20:** Trong đa số trường hợp, interface và type alias hoán đổi được. 3 điểm khác biệt quan trọng.

**3 actual differences:**

| Feature | Interface | Type Alias |
|---|---|---|
| Declaration merging | ✅ Yes — define same name twice → merged | ❌ No — error |
| `implements` in class | ✅ Yes | ❌ No |
| Function types | Both OK (different syntax) | Both OK |
| Primitive aliasing | ❌ No (only object types) | ✅ Yes (`type ID = string`) |
| Union types | ❌ No | ✅ Yes (`type X = A \| B`) |

**Declaration merging use case:**
```typescript
interface Config { timeout: number }
interface Config { retries: number }  // merged! No error
// Result: Config has both timeout AND retries
```
Useful khi extend 3rd-party library interfaces.

**When to use which:**
- Default: either (personal preference)
- Need declaration merging (library extension): interface
- Need union types / primitive alias: type alias
- Classes implementing: interface

**Active Recall:**
1. Q: 3 practical differences giữa interface và type? → A: (1) Declaration merging (interface only); (2) `implements` (interface only); (3) Union/primitive types (type alias only)
2. Q: Declaration merging dùng khi nào? → A: Khi extend 3rd-party library interfaces không control được
3. Q: `type Admin = User | Admin` có dùng interface không? → A: Không — union types chỉ với `type` keyword

### L83 — Interfaces for Function Types (STUDY)

**80/20:** Interface có thể define function types (less common, `type` alias thường dùng hơn).

**Must-know:**
```typescript
// type alias way (common)
type SumFn = (a: number, b: number) => number

// interface way (less common)
interface SumFn {
  (a: number, b: number): number
}
```
- Cả 2 work — interface way ít phổ biến hơn
- Know it exists, prefer type alias for function types

**Active Recall:**
1. Q: Interface syntax cho function type? → A: `interface Fn { (param: Type): ReturnType }`
2. Q: Nên dùng interface hay type alias cho function types? → A: Type alias — phổ biến và clear hơn
3. Q: Function type interface có thể làm gì type alias không? → A: Không — trong trường hợp này hoàn toàn tương đương

### L84 — Implementing Interfaces (DEEP_DIVE)

**80/20:** `class User implements Authenticatable` = class PHẢI có tất cả properties/methods trong interface.

**Must-know:**
```typescript
class AuthUser implements Authenticatable {
  constructor(public email: string, public password: string) {}
  login() { /* ... */ }
  logout() { /* ... */ }
  // can have MORE props/methods than interface requires
  userName: string = ""
}
```
- `implements` = TS-only
- Có thể implement MULTIPLE interfaces: `implements A, B, C`
- Class phải có ÍT NHẤT những gì interface requires

**Active Recall:**
1. Q: `implements` vs `extends` — khác nhau? → A: `extends` = inherit from parent class (JS feature); `implements` = enforce interface contract (TS-only)
2. Q: Một class có thể implement nhiều interfaces không? → A: Có — `class Foo implements A, B, C`
3. Q: Class có thể có thêm props ngoài interface yêu cầu không? → A: Có — phải có ÍT NHẤT interface's shape, có thể có nhiều hơn

### L85 — Using Interface as Parameter Type (STUDY)

**80/20:** Dùng interface làm type cho function parameter = guaranteed minimum shape.

**Must-know:**
```typescript
function authenticate(user: Authenticatable) {
  user.login()  // guaranteed to exist
}
// Any class implementing Authenticatable can be passed
authenticate(new AuthUser(...))
```
- Interface là type → function chỉ cần biết minimum contract
- Useful cho polymorphism — nhiều classes khác nhau có thể pass

**Active Recall:**
1. Q: Tại sao dùng interface thay vì concrete class làm type? → A: Flexibility — nhiều classes implement interface có thể pass; loose coupling
2. Q: Sau `function foo(user: Authenticatable)` — có thể access `user.login()` không? → A: Có — guaranteed by interface contract
3. Q: Là `optional properties` trong interface dùng `?` không? → A: Có — `email?: string` work trong interface

### L86 — Extending Interfaces (DEEP_DIVE)

**80/20:** `interface Admin extends Authenticatable` = inherit interface properties + add more. Không touch original.

**Must-know:**
```typescript
interface Admin extends Authenticatable {
  role: "admin" | "superadmin"
}
```
- `extends` work với interfaces (như `extends` với classes)
- Tạo new interface building on existing one
- Khác declaration merging: này creates NEW interface

**Active Recall:**
1. Q: Khác nhau `extends` interface và declaration merging? → A: `extends` = new interface, original unchanged; merging = adds to same interface
2. Q: Interface có thể extends nhiều interfaces không? → A: Có — `interface C extends A, B`
3. Q: Class `extends` và interface `extends` — khác nhau? → A: Class extends: only 1 parent (single inheritance); Interface extends: multiple OK

## Summary

19 lectures hoàn thành. Core mental model nắm được:

**Classes:**
- TS adds: explicit field declarations, access modifiers (public/private/protected/readonly), constructor shortcut, abstract keyword
- JS features still work: extends, super(), static
- `abstract` class = cannot instantiate, must be extended; abstract methods must be implemented

**Interfaces:**
- TS-only: compiled away completely
- 2 main uses: (1) object type definition; (2) class contract with `implements`
- vs Type Alias: mostly interchangeable; interface wins for declaration merging + `implements`; type alias wins for unions + primitives

**The 3 Critical Differences — Interface vs Type:**
1. Declaration merging: only interface
2. `implements` on class: only interface  
3. Union types / primitive alias: only type alias

**Access Modifiers cheat sheet:**
```
public    = accessible everywhere (default)
private   = this class ONLY
protected = this class + subclasses
readonly  = can read, cannot reassign
static    = belongs to class, not instance
abstract  = no instantiation; abstract methods must be overridden
```
