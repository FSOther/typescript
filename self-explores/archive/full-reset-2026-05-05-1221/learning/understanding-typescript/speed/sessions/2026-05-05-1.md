### L13 — Working with Types (STUDY)
**80/20:** TypeScript adds explicit type annotations (`: string`) on top of JS. Annotations stripped at compile time — pure syntax layer.

**Must-know:**
- `let username: string` = type annotation
- Lowercase: `string`, `number`, `boolean` (NOT `String`)
- Types = TS-only; type values = JS too (typeof operator)

**Active Recall:**
1. Q: Annotation được viết ở đâu? → A: Sau variable name, trước `=`: `let x: string`
2. Q: Compiled JS có giữ type annotation không? → A: Không — bị strip hoàn toàn
3. Q: Tại sao dùng lowercase `string` không phải `String`? → A: `string` là built-in TS type; `String` là JS constructor wrapper (khác nhau)

### L14 — Vanilla JS Has Types Too (SKIM)
**80/20:** JS có types nhưng không có explicit assignment. TS thêm enforcement tại compile time.

**Active Recall:**
1. Q: `typeof` là TS hay JS? → A: JS (JavaScript built-in operator)
2. Q: Khác biệt chính TS vs JS về types là gì? → A: TS cho phép *explicit* type assignment + type errors ở compile time
3. Q: Vì sao TS strip type annotations? → A: Browser/Node chỉ chạy JS, không hiểu TS syntax

### L15 — Type Inference vs Type Assignment (STUDY)
**80/20:** Nếu variable có initial value → TypeScript tự infer. Không cần annotate. Nếu không có initial value → phải annotate.

**Must-know:**
- `let age = 36` → TypeScript infers `number`, không cần `: number`
- `let username: string` → phải annotate (no initial value)
- Best practice: đừng annotate nếu TS có thể infer

**Active Recall:**
1. Q: Khi nào cần explicit type annotation? → A: Khi variable không có initial value
2. Q: `let x = "hello"` — có cần `: string` không? → A: Không — TS infers từ initial value
3. Q: Nếu inferred type không đúng, làm sao override? → A: Thêm explicit annotation: `let x: string | number = "hello"`

### L16 — Function Parameter Types (STUDY)
**80/20:** Parameters nên luôn có explicit type (vì không có "initial value" để infer). Return type thường inferred.

**Must-know:**
- `function add(a: number, b: number)` = parameter types
- `function add(a: number, b = 5)` = b inferred as number từ default value
- Cùng syntax, colon sau param name

**Active Recall:**
1. Q: Tại sao parameters hay cần explicit types hơn variables? → A: Không có "initial value" để infer từ
2. Q: Viết function với default param và type inference: `function greet(name: string, times = 3)` — type của `times` là gì? → A: `number` (inferred từ default value 3)
3. Q: Có bắt buộc annotate params không? → A: Không bắt buộc, nhưng nếu không có default thì TS infers `any` — xấu

### L17 — The `any` Type (STUDY)
**80/20:** `any` = thoát khỏi TypeScript. Dùng như last resort. Prefer union types.

**Must-know:**
- `let age: any` → accepts string, number, boolean, anything
- Defeats purpose of TypeScript
- Use only when truly can't be more specific

**Active Recall:**
1. Q: Tại sao `any` là bad practice? → A: Disables type checking — essentially back to vanilla JS
2. Q: Alternative tốt hơn `any` khi cần flexibility là gì? → A: Union types: `string | number`
3. Q: Khi nào nên dùng `any`? → A: Truly last resort — khi không thể more specific và không có better alternative

### L18 — Union Types (STUDY)
**80/20:** `|` để combine types. `string | number` = accept either. More specific than `any`.

**Must-know:**
- `let age: string | number` = accepts string OR number
- `boolean | null | string` = 3+ types in union
- TypeScript still rejects anything not in the union

**Active Recall:**
1. Q: Syntax của union type là gì? → A: `TypeA | TypeB` với pipe symbol
2. Q: `let x: string | number = true` → lỗi gì? → A: `boolean` not assignable to `string | number`
3. Q: Khi nào dùng union type thay vì single type? → A: Khi value hợp lệ có thể là một trong nhiều types (không phải any)

### L19 — Array Types (STUDY)
**80/20:** Array type = `ElementType[]`. TypeScript hiểu array elements phải đúng type.

**Must-know:**
- `string[]` = array of strings (inferred từ `["sports", "cooking"]`)
- `(string | number)[]` = array of strings OR numbers
- Errors khi push wrong type: `hobbies.push(42)` nếu `hobbies: string[]`

**Active Recall:**
1. Q: Type annotation cho array of numbers? → A: `number[]`
2. Q: Array chấp nhận string hoặc number? → A: `(string | number)[]` — cần wrap trong `()`
3. Q: TypeScript có infer array type không? → A: Có — từ initial value `["a", "b"]` → infers `string[]`

### L20 — Advanced Array Types (STUDY)
**80/20:** `Array<string>` = cách viết khác của `string[]`. Đây là *generic type* — sẽ học kỹ hơn sau.

**Must-know:**
- `Array<string>` ≡ `string[]` — 100% equivalent
- `Array<string | number>` ≡ `(string | number)[]`
- Angle brackets `<>` = generic type syntax

**Active Recall:**
1. Q: `string[]` và `Array<string>` — khác nhau không? → A: Không — hoàn toàn tương đương
2. Q: Generic type có dấu hiệu nhận biết gì? → A: Angle brackets `<>` sau type name
3. Q: Khi nào dùng `Array<T>` vs `T[]`? → A: Cả hai đều OK — style preference; `T[]` phổ biến hơn

### L22 — Tuples (STUDY)
**80/20:** Tuple = fixed-length array với defined types PER POSITION. `[number, string]` ≠ `(number | string)[]`.

**Must-know:**
- `let point: [number, number] = [1, 2]` → exactly 2 numbers
- Position matters: first must be first type, second must be second type
- Useful when length + types per position are known

**Active Recall:**
1. Q: Khác nhau giữa `[number, string]` và `(number | string)[]`? → A: Tuple: exactly 2 elements, position-typed; Array: any length, each can be number OR string
2. Q: `let result: [1 | -1, 1 | -1]` — được gán `[5, -1]` không? → A: Không — 5 is not `1 | -1`
3. Q: Khi nào dùng tuple? → A: Khi biết chính xác số lượng elements và type của từng position

### L26 — Enums (STUDY) — CRITICAL
**80/20:** `enum` = named set of values. Compiled to JS object (numbers). `Role.Admin = 0` mặc định. PITFALL: under the hood là numbers.

**Must-know:**
```typescript
enum Role { Admin, Editor, Guest }  // Admin=0, Editor=1, Guest=2
let userRole: Role = Role.Admin     // stores 0
```
- Có thể override: `Admin = 1` → 1, 2, 3
- String enums: `Admin = "admin"` nhưng phải gán ALL values
- Compiled to JS object → không native JS feature

**Pitfall:** `enum Role { Admin = 1 }` — `userRole = 0` → ERROR vì không có option nào = 0 nữa

**Active Recall:**
1. Q: Default value của enum options là gì? → A: Numbers bắt đầu từ 0 (0, 1, 2, ...)
2. Q: `enum Color { Red, Green, Blue }` — `Color.Green` bằng gì? → A: `1`
3. Q: Tại sao enum là CRITICAL pitfall? → A: Under the hood = numbers → nếu thay đổi enum order/values, runtime behavior thay đổi silently

### L34 — Type Narrowing (STUDY)
**80/20:** TypeScript tracks type through code flow. After null check → type narrowed. `typeof` for narrowing primitives.

**Must-know:**
- If check narrows: `if (!el) throw...` → after this, `el` is not null
- `typeof val === "string"` → val is `string` in that branch
- `in` operator: `"log" in val` → val has property `log`

**Active Recall:**
1. Q: Type narrowing là gì? → A: TS thu hẹp type dựa trên code flow (if checks, typeof, etc.)
2. Q: `typeof` là JS hay TS? → A: JavaScript — `typeof x === "string"` etc.
3. Q: Sau `if (typeof val === "number") { ... }` — type của val TRONG block là gì? → A: `number`

### L35 — `!` và `?.` Operators (STUDY)
**80/20:** `!` = "I promise not null" (risky). `?.` = "access only if not null" (safe, JS feature).

**Must-know:**
- `getElementById("id")!` → non-null assertion, trusts developer
- `inputEl?.value` → returns undefined if null, no error
- `!` dangerous, `?.` safe

**Active Recall:**
1. Q: `document.getElementById("btn")!` — `!` làm gì? → A: Non-null assertion — tells TS result is definitely not null
2. Q: `inputEl?.value` vs `inputEl.value` — khác nhau? → A: `?.` = safe — returns undefined if null; `.` = runtime error if null
3. Q: `!` là TS-specific không? → A: Có — non-null assertion là TS feature; `?.` là JS feature

### L37 — `unknown` Type (STUDY)
**80/20:** `unknown` = như `any` nhưng SAFE — buộc phải narrow trước khi dùng. Dùng cho truly unknown values.

**Must-know:**
- `let val: unknown` → phải check type trước khi dùng
- `if (typeof val === "string") { val.toUpperCase() }` → OK
- `val.toUpperCase()` without check → TS ERROR (unlike `any`)

**Active Recall:**
1. Q: Khác nhau `any` và `unknown`? → A: `any` = no restrictions; `unknown` = must narrow type before use
2. Q: Khi nào dùng `unknown` vs `any`? → A: `unknown` khi thực sự không biết type nhưng muốn type-safe; `any` = never if possible
3. Q: `let x: unknown; x.toUpperCase()` → lỗi không? → A: Lỗi — phải narrow trước: `if (typeof x === "string") x.toUpperCase()`

### L38 — Optional Parameters/Properties (STUDY)
**80/20:** `?` sau param/property name = optional. Có thể omit khi gọi hàm hoặc tạo object.

**Must-know:**
```typescript
function foo(msg?: string) {}  // can call foo() or foo("hi")
type User = { name: string; role?: "admin" | "guest" }
```
- `?` đặt TRƯỚC `:` type annotation
- Type becomes `T | undefined` implicitly

**Active Recall:**
1. Q: Syntax của optional parameter? → A: `paramName?:` — question mark BEFORE colon
2. Q: Optional property trong object type? → A: `{ name?: string }` — có thể omit khi tạo object
3. Q: `function foo(x?: number)` — type của `x` inside function là gì? → A: `number | undefined`

---

### L39 — Nullish Coalescing `??` (STUDY)
**80/20:** `??` = fallback ONLY cho null/undefined. Khác `||` (fallback cho falsy). JS feature.

**Must-know:**
```typescript
const val = input ?? "default"  // "default" ONLY if input is null/undefined
const val = input || "default"  // "default" if input is falsy (includes "", 0, false)
```
- `""` → `||` uses fallback; `??` does NOT (empty string is not null/undefined)
- JS feature, not TS-specific

**Active Recall:**
1. Q: `null ?? "fallback"` → kết quả? → A: `"fallback"`
2. Q: `"" ?? "fallback"` → kết quả? → A: `""` — vì `""` không phải null/undefined
3. Q: Dùng `??` hay `||` khi muốn preserve empty string? → A: `??` — vì `||` sẽ replace empty string với fallback

## Summary

27 lectures hoàn thành. Core mental model nắm được:

**TypeScript = JavaScript + Type Labels**
- Type annotations stripped at compile time
- Type inference handles most cases (don't over-annotate)
- Type system: `string | null`, `string[]`, tuples, enums, literal types, Record, unknown

**Critical pitfalls:**
- `{}` as type ≠ empty object (= any non-null)
- Enums compile to numbers → reverse mapping pitfall
- `!` (non-null assertion) disables safety check
- `any` defeats TS purpose → use `unknown` or union types

**Operators to memorize:**
- `|` = union types
- `?` = optional (params/properties)
- `!` = non-null assertion (dangerous)
- `?.` = optional chaining (safe, JS)
- `??` = nullish coalescing (null/undefined only, JS)
- `as` = type casting (take responsibility)
