### L89 — Intersection Types (DEEP_DIVE)

**80/20:** `&` combines object types (all props from both). Alternative to `extends` in interfaces.

**Must-know:**
```typescript
type FileData = { path: string; content: string }
type Status = { isOpen: boolean; errorMessage?: string }
type AccessedFileData = FileData & Status  // has all 4 props
```
- `&` (ampersand) = intersection type
- Same effect as interface `extends` — choose based on style
- Can chain: `A & B & C`

**vs Union (`|`):**
- Union `A | B` = one OR the other
- Intersection `A & B` = both combined (all properties)

**Active Recall:**
1. Q: `type AB = A & B` — AB có props của A không? → A: Có — AB có ALL props của cả A và B
2. Q: Interface version của intersection là gì? → A: `interface AB extends A, B { }`
3. Q: `string & number` có nghĩa gì? → A: `never` — không có value nào vừa là string vừa là number

### L90 — Type Guards (DEEP_DIVE)

**80/20:** Type guard = if check để narrow type. `in` operator phổ biến cho union object types.

**Must-know:**
```typescript
type Source = FileSource | DBSource  // FileSource has .path, DBSource has .connectionURL

function load(source: Source) {
  if ("path" in source) {
    // TypeScript knows: source is FileSource here
    return source.path  // OK
  }
  // TypeScript knows: source is DBSource here
  return source.connectionURL  // OK
}
```
- `in` operator = check property existence
- TypeScript narrows type inside if block

**Active Recall:**
1. Q: Type guard là gì? → A: If check để narrow union type → TypeScript biết concrete type trong từng branch
2. Q: `"path" in source` làm gì? → A: Checks nếu source có property "path" — if yes, TypeScript narrows to FileSource
3. Q: Sau `if ("path" in source) { return }` — TypeScript biết gì về source sau if? → A: Narrows to DBSource (FileSource case đã return)

### L91 — Discriminated Unions (DEEP_DIVE)

**80/20:** Thêm shared `type` property vào mỗi type trong union → guard bằng value of `type`. Cleaner than checking prop existence.

**Must-know:**
```typescript
type FileSource = { type: "file"; path: string }
type DBSource = { type: "db"; connectionURL: string }
type Source = FileSource | DBSource

function load(source: Source) {
  if (source.type === "file") {
    return source.path  // TypeScript knows it's FileSource
  }
  return source.connectionURL  // TypeScript knows it's DBSource
}
```
- Pattern: add `type: "file"` literal type as discriminant
- Switch statement works great with discriminated unions

**Full switch pattern:**
```typescript
switch (source.type) {
  case "file": return source.path
  case "db": return source.connectionURL
  default: const _exhaustive: never = source  // compile error if case missing
}
```

**Active Recall:**
1. Q: Discriminated union pattern là gì? → A: Thêm shared `type` property với literal type vào mỗi union member → guard bằng that property's value
2. Q: Tại sao dùng discriminated union thay `in` operator? → A: Property names có thể thay đổi; literal type discriminant stable hơn
3. Q: Exhaustiveness check với `never` là gì? → A: `default: const x: never = action` → compile error nếu switch không cover all cases

### L92 — `instanceof` Type Guard (DEEP_DIVE)

**80/20:** `instanceof` dùng khi narrowing classes. JS built-in operator.

**Must-know:**
```typescript
class User { join() {} }
class Admin { scan() {} }
type Entity = User | Admin

function init(entity: Entity) {
  if (entity instanceof User) {
    entity.join()  // TypeScript knows it's User
  } else {
    entity.scan()  // TypeScript knows it's Admin
  }
}
```
- `instanceof` = standard JavaScript, not TS-specific
- Works only with classes (not plain objects or type aliases)

**Active Recall:**
1. Q: `instanceof` hay `in` operator — khi nào dùng cái nào? → A: `instanceof` cho classes; `in` cho plain objects/interfaces
2. Q: `instanceof` là JS hay TS? → A: JavaScript — built-in operator
3. Q: `if (x instanceof User)` — TypeScript sẽ narrow type thành gì trong block? → A: `User`

### L93 — Type Predicates (STUDY)

**80/20:** Modern TS infers type predicates from simple if-return functions. Cho phép reuse type guard logic.

**Must-know:**
```typescript
function isFileSource(source: Source): source is FileSource {
  return source.type === "file"
}
// Modern TS can infer this return type automatically
```
- Type predicate return type: `paramName is TypeName`
- Function returning this predicate = reusable type guard

**Active Recall:**
1. Q: Type predicate syntax? → A: `function isX(val: T): val is X { return ... }`
2. Q: Tại sao cần type predicate? → A: Outsource type guard logic để reuse across multiple functions
3. Q: Modern TS có tự infer type predicates không? → A: Có — nhưng explicit annotation vẫn tốt cho clarity

### L94-L95 — Function Overloads (STUDY)

**80/20:** Nhiều function signatures cho 1 function body → TS biết return type per input type.

**Must-know:**
```typescript
function getLength(val: string): string  // overload 1
function getLength(val: any[]): number   // overload 2
function getLength(val: string | any[]) { // implementation
  if (typeof val === "string") return `${val.split(" ").length} words`
  return val.length
}

const result = getLength("hello world")  // TypeScript knows: string
const count = getLength([1, 2, 3])       // TypeScript knows: number
```
- Overload signatures go BEFORE implementation
- TS uses overload to determine return type per call site

**Active Recall:**
1. Q: Function overloads giải quyết vấn đề gì? → A: Khi function return type phụ thuộc vào input type — TS cần biết return type chính xác hơn generic union
2. Q: Overload signatures có body không? → A: Không — chỉ signature, body thuộc về implementation (cuối cùng)
3. Q: Cần mấy signatures minimum? → A: Ít nhất 1 overload + 1 implementation signature

### L96 — Index Types (STUDY)

**80/20:** `{ [key: string]: ValueType }` = object với dynamic keys. Flexible object type.

**Must-know:**
```typescript
type DataStore = {
  [prop: string]: number | boolean  // any string key, number or boolean value
}
const store: DataStore = {}
store.id = 5      // OK
store.isOpen = true  // OK
store.name = "Max"  // ERROR: string not assignable
```
- Placeholder in `[]` = any number of dynamic properties
- Key type: usually `string` (also `number`, `symbol`)

**Active Recall:**
1. Q: Index type syntax? → A: `{ [keyName: KeyType]: ValueType }` — square brackets with key type
2. Q: Khác nhau index type và `Record<K,V>`? → A: Functionally same — `Record<string, number>` = `{ [k: string]: number }`
3. Q: Có thể mix static + dynamic properties không? → A: Có — `{ id: number; [k: string]: number }` nhưng static type phải compatible với dynamic

### L97 — `as const` (STUDY)

**80/20:** `as const` = narrow inferred type to narrowest possible (literal types, readonly). Prevents widening.

**Must-know:**
```typescript
const roles = ["admin", "guest", "editor"] as const
// Type: readonly ["admin", "guest", "editor"]  ← not string[]
// roles[0] is "admin" not string
roles.push("new")  // ERROR: readonly
```
- Without `as const`: inferred as `string[]`
- With `as const`: inferred as `readonly ["admin", "guest", "editor"]` 

**Active Recall:**
1. Q: `["a", "b"] as const` — type là gì? → A: `readonly ["a", "b"]` (tuple of literal types, not string[])
2. Q: `as const` có mutation implications không? → A: Có — array becomes readonly, can't push
3. Q: Khi nào cần `as const`? → A: Khi muốn narrow type thay vì general (enum-like constants, fixed config)

### L99 — `satisfies` Keyword (STUDY)

**80/20:** `satisfies Type` = validate value MATCHES type, but keep NARROW inferred type. Rare but powerful.

**Must-know:**
```typescript
// With :
const data: Record<string, number> = { entry1: 1, entry2: 2 }
data.entry3  // OK (no error — Record allows any key)

// With satisfies
const data = { entry1: 1, entry2: 2 } satisfies Record<string, number>
data.entry3  // ERROR — TypeScript keeps narrow type with only entry1/entry2
```
- `:` = widened to declared type
- `satisfies` = validated against type, but TYPE kept from VALUE
- Use case: library code, config validation

**Active Recall:**
1. Q: `satisfies` vs `:` type annotation — key difference? → A: `:` widens to declared type; `satisfies` validates + keeps narrow inferred type
2. Q: Khi nào dùng `satisfies`? → A: Library code, route configs — khi muốn enforce type shape NHƯNG keep narrow inference
3. Q: Ít dùng `satisfies` trong practice không? → A: Có — advanced feature, specific use cases only

### L102-L103 — Creating Generic Types (STUDY)

**80/20:** Generic type = `type Name<T> = ...` dùng T như placeholder. Dùng `Name<ConcreteType>` khi sử dụng.

**Must-know:**
```typescript
type DataStore<T> = {
  [prop: string]: T
}
const stringStore: DataStore<string> = {}
```

**Active Recall:**
1. Q: Syntax tạo generic type? → A: `type Name<T> = ...` — T là placeholder
2. Q: Dùng generic type như thế nào? → A: `Name<ConcreteType>` — plug in concrete type between angle brackets
3. Q: Tại sao generic hơn nhiều type aliases? → A: 1 definition, infinite concrete types — không cần `DataStoreString`, `DataStoreBoolean`, etc.

### L104 — Generic Functions + Inference (STUDY)

**80/20:** Generic function = `function name<T>(arg: T): T`. TS infers T from call site — no need to explicit specify.

**Must-know:**
```typescript
function merge<T>(a: T, b: T): T[] {
  return [a, b]
}

const ids = merge(1, 2)    // T inferred as number
const names = merge("a", "b")  // T inferred as string
// merge<number>(1, 2)  // explicit but usually unnecessary
```
- Inference makes generic functions ergonomic
- T tracks type through function

**Active Recall:**
1. Q: Generic function syntax? → A: `function name<T>(param: T): ReturnType`
2. Q: Cần explicit `merge<number>(1,2)` không? → A: Không — TS infers từ arguments
3. Q: Tại sao generic function hơn `any`? → A: Generic PRESERVES type info; `any` LOSES it

### L105 — Multiple Generic Parameters (STUDY)

**80/20:** `<T, U>` = 2 independent type params. Different types for different params.

**Must-know:**
```typescript
function merge<T, U>(a: T, b: U): [T, U] {
  return [a, b]
}
merge(1, "hello")  // T = number, U = string — allowed!
```
- T and U = independent placeholders
- Add as many as needed, separated by comma

**Active Recall:**
1. Q: Multiple type params syntax? → A: `<T, U>` — comma-separated inside angle brackets
2. Q: `merge<T>(a: T, b: T)` vs `merge<T, U>(a: T, b: U)` — khác nhau? → A: First: a and b same type; Second: a and b can be different types
3. Q: Convention cho multiple type params? → A: T, U, V, K, V — single uppercase letters; K/V common for key/value

### L106-L107 — Generic Constraints (DEEP_DIVE) — CRITICAL

**80/20:** `<T extends SomeType>` = T phải extend SomeType. Constrain flexibility while keeping generics.

**Must-know:**
```typescript
function mergeObj<T extends object, U extends object>(a: T, b: U): T & U {
  return { ...a, ...b }
}

mergeObj({ name: "Max" }, { age: 36 })  // OK
mergeObj(1, 2)  // ERROR: number doesn't extend object
```
- `extends` in generics = constraint (different from class extends!)
- T `extends` X = T must be compatible with X (be X or subtype of X)
- Inferred return type: `T & U` intersection

**Active Recall:**
1. Q: Generic constraint syntax? → A: `<T extends ConstraintType>` — T phải extend (be compatible with) ConstraintType
2. Q: `<T extends object>` nghĩa là gì? → A: T phải là object — primitives (number, string) bị rejected
3. Q: `extends` trong generic vs `extends` trong class — khác nhau? → A: Generic extends = constraint (T phải là subtype); Class extends = inheritance

**80/20:** Classes và interfaces cũng có thể generic. Same syntax: `class Name<T>`.

**Must-know:**
```typescript
class User<T> {
  constructor(public id: T) {}

const stringUser = new User<string>("abc")
const numUser = new User(123)  // T inferred as number
```

**Active Recall:**
1. Q: Generic class syntax? → A: `class Name<T> { ... }` — T usable trong class body
2. Q: Generic interface syntax? → A: `interface Name<T> { method(val: T): void }`
3. Q: TS infer T khi create generic class instance không? → A: Có — `new User(123)` infers T = number

Type Guards cheat sheet:
| Pattern | When | Example |
|---|---|---|
| `in` operator | Plain objects / interfaces | `"path" in source` |
| `instanceof` | Classes | `entity instanceof User` |
| Discriminated union | Shared `type` property | `source.type === "file"` |
| Type predicate | Reusable guard function | `function isX(v): v is X` |

**M08 — Generics:**

Generics cheat sheet:
```typescript
// Type
type Store<T> = { [k: string]: T }
const s: Store<number> = {}

// Function
function merge<T, U>(a: T, b: U): [T, U] { return [a, b] }

// Constraint
function mergeObj<T extends object, U extends object>(a: T, b: U): T & U {
  return { ...a, ...b }
}

// Class
class Box<T> { constructor(public value: T) {} }
```

**Key insight:** Generics = flexible types where you DON'T LOSE type information (unlike `any`). The type "flows through" the function/class.

**Critical distinction:**
- `extends` in class = inheritance
- `extends` in generic = constraint
