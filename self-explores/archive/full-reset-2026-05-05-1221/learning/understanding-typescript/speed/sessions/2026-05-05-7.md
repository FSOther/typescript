## Learning Output

### M03 — tsconfig.json (L41-L49)

#### 80/20 Summary

**5 điều PHẢI biết về tsconfig:**

1. **Tạo:** `tsc --init` → sinh ra `tsconfig.json`, sau đó `tsc` (không target file) để compile cả project
2. **`target`** — compile TS sang JS version nào (ES2016, ES2022, ESNext). Browser cũ → ES2016; Node/modern → ES2022+
3. **`strict: true`** — bật hết các type checks nghiêm ngặt. LUÔN để `true` trong project thực
4. **`outDir`** — nơi đặt file JS đã compile (thường là `dist/`). Kết hợp với `rootDir: "src"` để tách source và output
5. **`@types` packages** — TS không tự biết Node.js hay third-party lib types → cài `@types/node`, `@types/react` etc. như devDependency

#### Must-Know Terms

| Term | Ý nghĩa |
|---|---|
| `tsconfig.json` | File cấu hình TS — kiểm soát compile target, strictness, input/output |
| `target` | JS version output: ES2016/ES2022/ESNext |
| `strict` | Gộp nhiều strict checks: noImplicitAny, strictNullChecks, etc. |
| `outDir` | Thư mục output cho compiled JS files |
| `rootDir` | Thư mục source TS files |
| `@types/*` | Package cung cấp type declarations cho lib không built-in trong TS |
| DefinitelyTyped | GitHub repo quản lý tất cả `@types` packages |
| `noEmitOnError` | Không tạo JS output nếu có compile error |
| `sourceMap` | Tạo `.map` file để debug TS trong browser/DevTools |
| `noUnusedLocals` | Cảnh báo khi có biến/param không dùng |

#### Why It Matters

`tsconfig.json` là trái tim cấu hình mọi TS project — kiểm soát độ nghiêm ngặt type checking và cách compile. `@types` packages là cách TS "học" types của Node.js, React, lodash v.v. mà không cần đọc code thực của chúng.

#### What To Skip

- Hầu hết các options nâng cao (monorepo, `disableReferenceProjectLoad`, `paths`, module resolution)  
- `lib` array (chỉ cần khi custom, mặc định là đủ)  
- Bundler-related options (Vite/webpack sẽ tự config)

#### Active Recall

1. Lệnh nào để tạo tsconfig.json?
   → `tsc --init`

2. Tại sao cần cài `@types/node` khi dùng Node.js với TS?
   → Node.js types không built-in trong TS, cần package riêng để IDE và compiler biết types của fs, path, v.v.

3. Sự khác biệt giữa `strict: true` vs `noImplicitAny: true`?
   → `strict: true` bật nhiều checks cùng lúc (bao gồm `noImplicitAny`, `strictNullChecks`, v.v.). Dùng `strict: true` là đủ, không cần set từng cái.

4. Khi nào dùng `noEmitOnError: true`?
   → Luôn dùng trong project production — đảm bảo không tạo broken JS output khi có type errors.

### M04 — Mini Project (L50-L57)

#### 80/20 Summary

**Pattern của project TS thực tế:**

```typescript
// 1. Define input type (object shape)
type InvestmentData = {
  initialAmount: number;
  annualContribution: number;
  expectedReturn: number;
  duration: number;
};

// 2. Define output type
type InvestmentResult = {
  year: number;
  interest: number;
  total: number;
};

// 3. Function với explicit types
type CalculationResult = InvestmentResult[] | string;

function calculateInvestment(data: InvestmentData): CalculationResult {
  const { initialAmount, annualContribution, expectedReturn, duration } = data;
  // ...
}

function printResults(results: CalculationResult): void {
  // ...
}
```

**Key insights từ project này:**
- Reuse types — `CalculationResult` dùng cho cả return type của `calculateInvestment` lẫn param type của `printResults`
- Object type cho input → dễ extend sau, không phải sửa function signature
- Union return type (`InvestmentResult[] | string`) cho phép trả về error message hoặc data

#### Why It Matters

Đây là pattern chuẩn của TS real-world code: define types trước, viết function sau. Types như contract — IDE và compiler đảm bảo không ai vi phạm contract.

#### What To Skip

- Toàn bộ code chi tiết (đã nắm pattern là đủ)
- L50-L51 (intro + Node.js setup — biết rồi)

#### Active Recall

1. Tại sao nên dùng object type cho function params thay vì nhiều params riêng lẻ?
   → Dễ mở rộng, dễ destructure, code dễ đọc hơn khi có 4+ params

2. Union return type `InvestmentResult[] | string` có ý nghĩa gì?
   → Function có thể trả về array kết quả (thành công) hoặc error message string (thất bại)

3. Khi reuse type giữa nhiều functions, lợi ích là gì?
   → Nếu type thay đổi, chỉ cần sửa 1 nơi, tất cả chỗ dùng tự update — đây là nguyên lý DRY với types
